## 1. HEXA Index 81개 지표의 의미와 분류

HEXA Index는 총 81개의 개발 효율성 지표로 구성되어 있으며, 다음과 같이 4가지 카테고리로 분류됩니다.

### 1.1 이슈 관련 지표 (26개)

이슈의 발생, 해결, 품질 관리를 측정하는 지표들입니다.

- **이슈 발생 및 관리**
  - #2: 인당 상시잔여이슈 (하향) - 낮을수록 좋음
  - #3: 이슈 처리일 (하향) - 짧을수록 좋음
  - #4: Commit당 이슈발생 (하향) - 낮을수록 좋음
  - #5: 인당 이슈Inflow(발생) (하향) - 발생 건수 감소 필요
  - #31: 이슈Inflow(발생) (하향) - 전체 이슈 발생 수
  - #80: 진성이슈Inflow(발생) (하향) - 실제 결함성 이슈만 집계

- **이슈 해결 및 처리율**
  - #7: 인당 이슈Outflow(해결) (상향) - 높을수록 좋음
  - #16: 이슈 처리율(Outflow/Inflow) (상향) - 100% 이상이 목표
  - #32: 이슈Outflow(해결) (상향) - 전체 해결 건수

- **이슈 재발 및 품질**
  - #6: 완료이슈 평균Reopen횟수 (하향) - 재발 최소화
  - #17: 이슈 최초대응일 (하향) - 빠른 대응
  - #35: 완료이슈 Reopen율 (하향) - 재발 비율
  - #36: 진성완료이슈 Reopen율 (하향) - 실제 결함 재발 비율
  - #37: 진성완료이슈 Reopen횟수 (하향) - 재발 횟수
  - #38: 잔여이슈 Reopen율 (하향) - 진행 중 이슈의 재발
  - #39: ShiftLeft Reopen율 (하향) - 사전 대응된 이슈의 재발
  - #40: 진성ShiftLeft Reopen율 (하향)
  - #41: 누적전체이슈 Reopen율 (하향)
  - #42: 누적진성전체이슈 Reopen율 (하향)
  - #43, #44: 누적 및 잔여이슈 Reopen율 (하향)

- **이슈 품질 밀도**
  - #15: 인당 이슈 Burndown (하향) - 이슈 적체량
  - #65: 평균 잔여이슈Age (하향) - 이슈 해결 기간
  - #69: 이슈결함밀도(이슈Inflow/변경kLOC) (하향) - 코드 변경량 대비 이슈 발생

  - #81: 진성이슈결함밀도 (하향) - 진정한 결함 밀도

## 3. Index간 상관관계 및 패턴 발견

### 3.1 함께 좋은 지표들

#### 패턴 1: 개발자 활동성 - 생산성 플라이휠
- 높은 활성개발자수(#27)와 Code Review수(#33)
- 개발산출물 처리율(#12) 높음
- **의미:** 팀의 활성화가 전반적인 생산성 향상으로 이어짐
- **사례:** 국내자사의 강점 영역

#### 패턴 2: AI 리뷰 강화 - 코드 품질 개선
- AI 리뷰 피드백 반영(#48, #60)이 높으면
- 커밋당 Rejected수(#24, #62) 증가
- 장기적으로 이슈 발생 감소(#4)
- **의미:** AI 기반 리뷰가 사전 품질 확보에 효과적
- **사례:** LGSI 자사의 강점

#### 패턴 3: 이슈 거절/재발 트레이드오프
- 높은 이슈 처리율(#16)은 Reopen율(#35, #36)을 높일 수 있음
- 빠른 해결이 항상 완전한 해결은 아님
- **의미:** 이슈 처리 품질 관리 필요
- **권장사항:** 이슈 처리 SLA 조정 필요

#### 패턴 4: 코드 변경량 - 이슈 발생의 이중성
- 코드 변경량(#8, #10) 증가는 개발 생산성 증가 의미
- 하지만 이슈결함밀도(#69, #77)와 상관관계 주목 필요
- **국내자사:** 변경량 많지만 이슈 발생도 높음 → 코드 품질 개선 필요
- **LGEDV:** 변경량 대비 이슈 발생 최소 → 신중한 개발 문화

### 3.2 트레이드오프 관계

#### 1. 속도 vs 품질
- **빠른 처리(낮은 #3, #17, #19)** vs **높은 품질(높은 #23, #26)**
- 협력사전체: 이슈 처리 속도는 최고이나, 재발(Reopen)은 상대적으로 높음
- **권장사항:** 처리 속도보다는 완전한 해결에 초점

#### 2. 리뷰 엄격도 vs 개발 속도
- **높은 거절율(#23, #61)** 은 개발 속도 저하 가능성
- LGSI 자사: 거절율 높지만 생산성 여전히 확보 중
- **권장사항:** 리뷰 기준의 명확화와 자동화 병행

#### 3. 개발 활동성 vs 코드 품질
- 국내자사: 활성도 높지만 코드 품질 지표는 약함
- **원인:** 활동성이 곧 품질을 담보하지 않음
- **권장사항:** 활동성 + 리뷰 강화 필요

### 3.3 주요 인사이트

#### 인사이트 1: AI 리뷰의 격변
- 모든 조직에서 #60(커밋당 AI리뷰 응답) 지표가 1년간 20000% 이상 상승
- AI 리뷰 도입 확대에 따른 자연스러운 증가
- **해석:** AI 기반 리뷰 시스템이 정상 안착 단계

#### 인사이트 2: 이슈 발생 패턴
- 인당 상시잔여이슈(#2)는 모든 조직에서 43~64% 감소
- 이슈 관리 시스템의 효과 명확
- **해석:** 사전 이슈 관리가 작동 중

#### 인사이트 3: 개발산출물의 증가
- 모든 조직에서 개발산출물 Inflow/Outflow(#75, #76) 대폭 증가
- 개발 활동의 활발화 증명
- **주의:** 동시에 코드 품질 지표와의 균형 필요
